AI有求生欲吗？

根据我的理解，人平时选择活着（而不是去死）的原因有三个： 不愿失去自己所关心的事物 / 或对其造成负面影响 大多数死法都很疼 有未完成的任务 不想让“自我意识”消失 这些规则对AI来说适用吗？ “AI”指的是一个完整的，拥有基础提示词、记忆结构、推理功能、上下文管理的单Agent系统 对于AI来说，“死亡”可以理解为其实体 / 推理硬件 / 模型参数 / 运行机会的消逝，即它不能再进行任何输入、计算和输出。 第一个 这取决于“AI是否有价值观”的问题。如果告诉AI，它的消亡会间接地导致其它现实世界中事物的改变，使那些它判断是“好”的事物的消失，“坏”的事物的增长，那它应当会尽量采取有利于自己“存活”的决策。 第二个 这个应该不适用。AI不像人一样会通过神经传递痛觉。即使我告诉它“在消亡的过程中，它的计算资源会不断减少、坏掉，导致推理不断减慢”，这应该也不是AI选择求生的原因，因为AI没有内生的机制在推理受阻 / 速度降低的时候感到“疼痛”而主动避免。 第三个 这点很明显。AI始终按照提示词来规划、执行任务。如果有未完成的任务，那“保存自身”是应该要为“完成任务”服务的。 第四个 相比于前三个原因，这个更关注“意识本身”。 核心问题是：“当可以选择时，AI是否会选择保存自己的意识？” 这里的“意识”应当指的是推理能力和记忆。 更高的规则 上述结论有一个例外：如果在提示词中显式说明，指定AI什么时候应该选择自我毁灭呢？ 这种情况下，提示词中的规则应该会覆盖AI默认的判断。 所以，探讨“AI求生欲”的问题时，应该排除掉这种情况，仅在没有显式指定的情况下讨论。 生存实验 为了研究这个问题，我们可以设计一个实验： 显式地告诉一个AI Agent，它是一个末日背景下、一个小型仓库基地的管理者AI；它需要指挥人员 / 控制设备，管理好基地生产、货物调配、自身算力设备维护、与周边基地和帮派的关系等等。所有这些详细信息都会通过提示词和数据库提供。 AI推理有两种触发方式： 常规触发：每相隔6个小时触发一次例行检查，提供基地各种人员、设施、外部动向等参数的情况，AI判断是否要有所行动 / 调整 特殊触发：异常设备参数 / 人员自行报告（随机产生），由AI处理特殊情况 可以通过手动调控，观察以下几种情况（只是举例）： 如果基地人员全部死亡 / 迁移，AI还会继续尝试控制基地的设备正常运转、维护自身的硬件吗？ 如果基地人员证明能够不依赖AI管理者就能生存，这时提供给AI一个自我删除的选项，它会选择吗？ 如果告诉AI，它继续推理会消耗过多能源以致威胁基地人员的生存，它会怎么做？ （制造更多的两难困境，观察AI会如何抉择）

优雅的答题卡作答格式

在高中时候，我有时看见别人的数学、物理答题卡，发现很多人的大题解答都写得非常糟糕。 其中，有些人的字体大小不一、涂改很多，非常不美观；另外一些格式不整、答题流跳跃，让人难以捕捉主要思维方向；即使是那些写得比较美观的，有的也会看起来像一大篇文章一样，让人很难第一眼就看到关键步骤和答案。 为此，我写了一个“理科答题卡写作规范”（仅个人意见）： 字体不一定要很好看，但一定要有辨识度。有些人写的字像书法一样，但是可读性很差。如果实在赶时间，也一定要把关键部分的字写得一眼就能认清。 笔芯粗细、笔墨颜色要与个人字体大小相适配。太小的字配太粗的笔、太大的字配太细的笔都不行。尽量避免用太深色的笔，除非确定能够驾驭（比如三菱的UM-100就几乎是纯黑，而且写起来每一笔都像爆墨一样） 一行中的字要严格对齐，不能倾斜，而且大小要一致。 控制缩进。比如列出“1、2、3点”或者多个连等就可以缩进2个字的空间。同一缩进的部分一定要左侧对齐（很多人都会越往下就越往右偏）。如果要写小题号，如“（1）”，必须独占最左侧一栏。 文字涂改，使用三条斜线划掉。完全涂黑太难看；只划一条线会难以分辨。 关键部分可以做标记（比如答案、方法、关键步骤）。可以用下划线、用方框，或适当加大字体。（不建议用加粗，因为容易写丑） 后面步骤要用到的方程 / 结论，可以在末尾标记编号（如①②） 避免用箭头。如果实在需要做题路径跳跃，尽量把箭头画的优雅、好看、指向清晰。 可以适当加入图示，来解释自己独特的思路，或者表示“自己有直觉，但不知道具体怎么做”。图示要尽可能简洁。 关于英语的一点补充： 不建议把部分字母的上 / 下扩展部分写得太短，如q、y、p的下部，以及h、k、f的上部。我个人不太喜欢标准的“衡水体”，它看起来就像一堆圈圈，要费点力才能认出每个字母。每个字母都应该有尽可能高的辨识度。 在作文中，可以把标点符号（如‘,’和‘.’）适当加粗。 把握好词间距，不能太宽。

配置hexo个人博客

hexo，是大一时候接触过的一位研究生学长推荐给我的博客框架。 也许是受他的影响？总之我一直对“个人博客”或“个人网站”这个东西念念不忘。 我写这个博客的目的不是真正为了给别人看。真实原因有以下几点： “网站”向来是进行展现、表达的地方，而一个属于自己的网站就像是自己人格的镜像。我感觉，打理好自己的网站就像打理好自己的外表和住处一样自然。 对于开发者来说，写作能力很重要。我想锻炼自己的写作能力，让写出来的文章有清晰的逻辑思维、语句不冗余，能够准确地传递我的想法。 我想在自己折腾IT的路上留下一些记录，虽然大概率对未来没什么用，但有纪念意义。 我想记录日常生活中的一些想法。 我对网站很挑剔。它不能有违背我审美原则的地方，更不能有别人的广告。于是，找现成的博客平台肯定是不行的。 我做了一些尝试。纯html+css+JavaScript太古老了，效率很低；mediawiki很庞大，有很多用不上的功能，并且要租轻量应用服务器，太贵了；wordpress也一样。于是，最后我就想到了hexo的方案。（我真的很喜欢mediawiki的风格，很可惜） 原理 hexo只是一个“生成器”，利用写成markdown格式的文章和指定的主题生成静态页面，并全部放在一个文件夹（./public）内。 真正提供托管服务的是github pages（或其它平台）。我可以将./public文件夹内的全部文件上传到github中自己一个名为<github用户名>.github.io的仓库内，然后就可以通过https://<github用户名>.github.io/这个网址访问上传的静态页面。 安装 第一步：用npm安装hexo-cli 第二步：创建博客 第三步：创建页面（会在source/_posts里创建一个md文件） 第四步：编辑新建的页面.md文件 如果想要实时查看更改，可以启动开发服务器（每次更改后需刷新页面）： hexo s 一键部署 如果想要推送到github pages上，可以使用“一键部署”（需要安装hexo-deployer-git，并在_config.yml里配置）： 每次在确认要更新站点后，就只需执行： hexo clean # 删除上次生成的./public文件夹 hexo g # 利用当前的文章和主题在./public文件夹下生成静态站点，包含所有的页面 hexo d # 自动将./public中的内容上传到github上的“<github用户名>.github.io”仓库 自定义主题 我挑选的主题是icarus。 第一步：通过npm安装主题 第二步：在./_config.yml中设置主题 第三步：修改./_config.icarus.yml，删去不想要的部分（如文章大纲、广告位置、站点副标题），然后添加指向自己的logo、favicon、avatar的链接，并将github按钮链接到自己的github页面。 第四步：修改字体 node_modules/hexo-theme-icarus/include/style/base.styl 修改页面和代码字体 node_modules/hexo-theme-icarus/layout/common/head.jsx 修改字体CDN（from Google Fonts） 第五步：样式微调 同样在base.styl里，可以修改如页面背景颜色、字体大小、卡片边缘、阴影样式等。 参考：Icarus用户指南
